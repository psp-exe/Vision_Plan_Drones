Neuro-Symbolic Autonomy: Fine-Tuning Vision Language Models for Continuous-Space Drone Navigation in Dynamic Warehouse Environments 1. Introduction: The Convergence of Vision and Planning in Logistics The modern logistics landscape is characterized by an increasing demand for velocity, accuracy, and density. As e-commerce accelerates, the physical architecture of warehouses has evolved from single-layer floor storage to high-density, multi-tier racking systems that exploit vertical space. This structural shift presents a fundamental challenge for automation. Traditional ground-based mobile robots (AGVs and AMRs), constrained to a 2D plane, are increasingly insufficient for inventory management tasks that require accessing inventory located ten meters or more above ground level. Unmanned Aerial Vehicles (UAVs) offer a compelling solution due to their unrestricted 3D mobility; however, their deployment is historically hindered by a critical "neuro-symbolic gap" in robotic control architectures. This gap exists between the continuous, noisy, and high-dimensional world of sensory perception and the discrete, logical world of automated planning. Classical planning systems, governed by formalisms such as the Planning Domain Definition Language (PDDL), excel at generating complex, long-horizon sequences of actions to achieve symbolic goals. Yet, these systems typically rely on a "closed world assumption," requiring a static, manually curated description of the environment. Conversely, modern deep learning—specifically Computer Vision (CV)—has achieved unprecedented capabilities in object detection and semantic segmentation but lacks the temporal reasoning required to execute multi-step missions effectively. This report creates a comprehensive technical blueprint for bridging this gap. We address the specific problem of fine-tuning a Vision Language Model (VLM) to function not merely as a captioning tool, but as a deterministic state estimator that generates PDDL problem files from visual inputs. The operational context is a GPS-denied warehouse environment where a drone must navigate without pre-defined waypoints. Unlike traditional approaches that discretize space into a graph of known nodes, the proposed architecture operates in continuous space. It leverages PDDL 2.1’s numeric fluents to handle dynamic coordinate generation, integrating battery constraints and 3D flight mechanics. 1.1 The Limitations of Graph-Based Navigation in Warehouses Historically, warehouse automation has relied on graph-based navigation. AGVs follow magnetic tape or traverse a virtual grid of QR codes embedded in the floor. 1  In these systems, the world is discretized: the robot moves from Node A to Node B. The "move" action in standard PDDL reflects this discretization: (move?robot?from?to). However, applying this paradigm to drones imposes severe limitations. A drone operating in a 3D volume has six degrees of freedom. Constraining it to a sparse graph of "air waypoints" negates its primary advantage—agility. If a package is placed slightly off-center on a shelf, or if a forklift obstructs a standard flight corridor, a graph-based planner often fails because the necessary detour or inspection point does not exist as a node in its static database. The user requirement to operate with  "no predefined way points or locations"  necessitates a shift from  Discrete State Space Planning  to  Hybrid Metric Planning . In this paradigm, locations are not static symbols (e.g., wp_1, wp_2) defined a priori. Instead, they are dynamic geometric entities generated in real-time by the perception system. When the drone's camera detects a target, the VLM must estimate its 3D coordinates relative to the warehouse frame and instantiate a new object in the PDDL problem file with those specific numeric values. This requires the PDDL domain to support  numeric fluents —variables that track continuous quantities like $x, y, z$ coordinates, battery levels, and velocity. 2 1.2 The Neuro-Symbolic Architecture The proposed system architecture synthesizes three distinct technological pillars: Perception (The VLM):  A Vision Language Model, such as PaliGemma or LLaVA, fine-tuned to act as a "Neural Parser." It ingests RGB imagery and natural language instructions, outputting a structured definition of the current state (objects, coordinates, spatial relationships) in a format compatible with PDDL. 4 Reasoning (The Planner):  A PDDL 2.1 compliant planner (e.g., POPF, OPTIC, or the PlanSys2 engine) that accepts the VLM-generated problem file. It generates a temporal plan involving durative actions like fly_to_coordinates or scan_inventory. 6 Execution (ROS 2):  A robotic control layer that translates high-level PDDL actions into low-level velocity commands (geometry_msgs/Twist) and trajectory setpoints, handling the real-time feedback loops required for stable flight. 8 The report proceeds by dissecting each of these components. Section 2 establishes the theoretical foundations of Numeric PDDL 2.1. Section 3 constructs the "perfect" domain file, rigorously handling the physics of drone flight and battery consumption. Section 4 details the fine-tuning methodology for the VLM, emphasizing the use of synthetic data and constrained decoding to ensure valid output. Section 5 elaborates on the perception pipeline, specifically the transformation of 2D bounding boxes into 3D coordinates. Finally, Section 6 outlines the ROS 2 integration strategy. 2. Theoretical Framework: PDDL 2.1 and Continuous Space Planning To satisfy the requirement of a warehouse environment with no predefined waypoints, we must utilize the advanced features of PDDL 2.1. Standard PDDL (STRIPS) is insufficient because it treats actions as instantaneous and discrete. A drone flight from a charging station to a shelf at $(x=50, y=10, z=5)$ is a process that unfolds over time and consumes continuous resources (energy). 2.1 Numeric Fluents and Dynamic State In classical planning, a robot’s location is a predicate: (at robot1 waypoint_A). In our continuous domain, location is defined by numeric functions. PDDL 2.1 introduces the (:functions...) block, which allows us to define variables that map object types to real numbers. 2 For a drone $d$ and a target object $t$, we define: $(x \ d), (y \ d), (z \ d)$: The drone's current position. $(tx \ t), (ty \ t), (tz \ t)$: The target's position. $(battery \ d)$: The current energy level. This shift enables  dynamic instance generation . The PDDL domain does not need to know  where  the targets are in advance. The planner simply receives a problem file stating: "There is an object $O$ at coordinates $(12.5, 3.2, 4.0)$." The planner then solves the differential constraints (or their linear approximations) required to move the drone from its current $(x,y,z)$ to the target's coordinates. 2.2 Durative Actions and Temporal Constraints Drone operations are inherently temporal. A flight action has a duration that depends on the distance traveled and the drone's velocity. PDDL 2.1 supports (:durative-actions), which allows the modeling of time-dependent constraints. 2 A durative action is characterized by three temporal annotations: at start : Conditions that must hold at the moment of initiation (e.g., sufficient battery to  start  engines). over all : Invariants that must be maintained throughout execution (e.g., the drone must remain airborne; the path must remain clear). at end : Effects that are realized upon completion (e.g., the drone arrives at the target coordinates). This temporal modeling is critical for battery management. We can define the duration of a flight action dynamically: $$duration = \frac{distance}{speed}$$ And we can model battery consumption as a continuous process using gradient effects (e.g., (decrease (battery?d) (* #t consumption_rate))), where #t represents the elapsed time. However, many planners (like POPF) handle discrete updates at the start and end more robustly than continuous gradients, so a discrete approximation at the action's end is often preferred for stability. 2 2.3 The "No Waypoint" Strategy: Late-Binding Symbols The core innovation in this approach is the concept of  Late-Binding Symbols . In a traditional workflow, a human maps the warehouse and hardcodes waypoints. In our workflow, the "waypoint" is created only when the VLM sees a target. The VLM performs a "referring expression comprehension" task. Given the instruction "Inspect the red box," it detects the box, calculates its 3D position, and generates a PDDL object red_box_01 with associated coordinates. The planner then treats red_box_01 as a valid destination. This allows the drone to navigate to  any  point in the continuous volume of the warehouse, unrestricted by a pre-computed graph. 3. Developing the Drone Warehouse Domain The domain file is the logical kernel of the system. It defines the physics of the world and the capabilities of the agent. Below, we construct a PDDL 2.1 domain optimized for a drone in a warehouse, specifically addressing the user's constraints: continuous space, 3D movement, battery management, and inventory inspection. 3.1 Domain Requirements Analysis To create a "perfect" domain file, we must model the following physical realities: 3D Mobility:  The drone moves in $x, y, z$. The domain must prevent ground collisions (flying requires being airborne). Resource Constraints:  Flying consumes battery per meter; hovering (scanning) consumes battery per second. Action Dependencies:  The drone cannot scan a box unless it is physically close to it. It cannot take off if it is already flying. Metric Distance:  The cost of movement is proportional to Euclidean distance. 3.2 The Domain File: warehouse_drone_continuous.pddl The following PDDL code implements these requirements. It uses types, numeric fluents, and durative actions. The distance-to Function: Standard PDDL planners (like POPF) often lack support for complex arithmetic functions like square roots ($ \sqrt{(x_2-x_1)^2 +...} $) within the action duration calculation. Even if supported, calculating Euclidean distance dynamically inside the planner is computationally expensive and can cause search space explosions. Solution:  We offload the geometry calculation to the perception layer. When the VLM detects a target, the system's "Problem Expert" (a software node) calculates the Euclidean distance between the drone's current known position and the target's estimated position. It then writes this value into the PDDL problem file as a static initialization: (= (distance-to drone1 target_box) 15.4). This simplifies the planner's job to basic arithmetic (division for time, multiplication for battery). 2. Decoupled Coordinates: We explicitly model x, y, and z as separate functions. This is crucial for "No Waypoint" navigation. If we used a single symbol location_A, the drone would be tied to the graph. By using raw coordinates, the fly_to_target action effect (assign (x?d) (tx?t)) tells the planner: "After this action, the drone will be spatially coincident with the target." This allows the drone to fly to  any  float-value coordinate generated by the vision system. 3. Separation of Flying and Scanning: The fly_to_target action brings the drone to the vicinity of the object, while scan_target performs the task. The precondition (<= (distance-to?d?t) (scan-range)) enforces a proximity constraint. This allows the planner to sequence a "fly" action followed immediately by a "scan" action, correctly accounting for the energy costs of both transit and stationary hovering. 4. Fine-Tuning the Vision Language Model The domain file provides the laws of physics, but the  problem file  provides the current reality. The VLM's role is to act as a  deterministic compiler , translating an image and a user instruction into a structured PDDL problem file that instantiates the objects and coordinates required by the domain. 4.1 The Challenge: Syntax Sensitivity Generic VLMs (like GPT-4V or Gemini) are trained for chat. When asked to generate PDDL, they often hallucinate predicates (e.g., using (at-loc) when the domain defines (at-coords)) or miss closing parentheses. A single syntax error renders the PDDL file unparseable by the solver. 11  Therefore, we do not fine-tune the VLM to output raw PDDL text. Instead, we fine-tune it to output a rigid  JSON Schema . A deterministic post-processing script then converts this JSON into the valid .pddl format. This "Intermediate Representation" (IR) strategy significantly reduces the error rate. 4.2 Model Selection: PaliGemma and LLaVA For this application,  PaliGemma  (Google) is the recommended architecture. 12 Reasoning:  PaliGemma is a VLM explicitly designed for "referring expression segmentation" and object localization. It is pre-trained to output bounding box coordinates in a textual format <loc000><loc111>. This native understanding of spatial coordinates makes it superior to standard captioning models for warehouse navigation tasks where precise $x,y,z$ estimation is required. Alternative:   LLaVA-Next  (Large Language and Vision Assistant) is a viable alternative if the instruction complexity (reasoning chain) is high, as it has a stronger LLM backbone (Vicuna/Llama-3). 14 4.3 Dataset Curation: The "Sim-to-Real" Pipeline Collecting thousands of real-world annotated drone images with perfect 3D ground truth is prohibitively expensive. We leverage  Synthetic Data Generation  using simulators like NVIDIA Isaac Sim or Gazebo. 16 The Data Generation Workflow: Scene Generation:  Procedurally generate warehouse aisles in the simulator. Randomize shelf heights, box colors, lighting conditions, and drone positions. Ground Truth Capture:  For every frame, extract: The RGB Image. The 2D Bounding Box of target objects. The 3D World Coordinates of the targets. The Drone's current pose. Instruction Generation:  Use a template-based engine to create natural language prompts: "Inspect the red crate on the top shelf." "Go to the blue barrel in aisle 4." Label Generation:  Create the target JSON output. LoRA freezes the pre-trained weights and injects trainable rank-decomposition matrices into the Transformer layers. This prevents "catastrophic forgetting" of the model's general visual knowledge while specializing it for the PDDL task. 13 Training Hyperparameters: Rank ($r$):  16 (Sufficient for syntax adaptation). Alpha ($\alpha$):  32. Target Modules:  q_proj, v_proj (Attention queries and values). Loss Function:  Cross-Entropy on the JSON tokens. Epochs:  3-5 (Overfitting is a risk with synthetic data; early stopping is essential). 4.5 Structured Output Enforcement During inference (deployment), we employ  Grammar-Constrained Decoding  (available in frameworks like vLLM or Outlines). 19  We feed the VLM the JSON Schema of our expected output. The inference engine masks out any tokens that would violate the schema. Result:  The VLM is mathematically incapable of producing invalid JSON. It  must  output a structure containing objects, coords, and goal. 5. Perception Pipeline: From Pixels to PDDL Fine-tuning teaches the VLM to  identify  the object and structure the data. However, converting a 2D bounding box into a precise 3D coordinate for the PDDL file requires a geometric perception pipeline. 5.1 Monocular Depth Estimation The drone camera provides a 2D projection of the 3D world. To recover the $z$ (depth) dimension, we integrate a Monocular Depth Estimation network (e.g.,  MiDaS  or  Depth Anything ) into the pipeline. 21 The Pipeline: Detection:  The Fine-Tuned VLM outputs the bounding box $(u, v, w, h)$ of the target. Depth Map:  The depth network processes the RGB image to generate a dense depth map $D$. Depth Sampling:  We calculate the median depth value $d_{obj}$ within the bounding box region of $D$. $$d_{obj} = \text{median}(D[v:v+h, u:u+w])$$ 5.2 Coordinate Transformation (Pinhole Model) We transform the image coordinates $(u_c, v_c)$ (center of the box) and depth $d_{obj}$ into the camera's 3D coordinate frame using the  Pinhole Camera Model . 23 $$X_{cam} = \frac{(u_c - c_x) \cdot d_{obj}}{f_x} \\ Y_{cam} = \frac{(v_c - c_y) \cdot d_{obj}}{f_y} \\ Z_{cam} = d_{obj}$$ Where $f_x, f_y$ are the focal lengths and $c_x, c_y$ are the optical centers (intrinsic parameters). Finally, we transform these coordinates from the  Camera Frame  to the  Warehouse (World) Frame  using the drone's current pose (obtained via Visual Odometry or SLAM): $$P_{world} = R_{drone} \cdot P_{cam} + T_{drone}$$ This $P_{world} (x, y, z)$ is the numeric value written into the PDDL problem file: (= (tx target_01) 12.5) (= (ty target_01) 3.2) (= (tz target_01) 4.0) 6. Execution and ROS 2 Integration The final piece of the architecture is the execution engine. We utilize  ROS 2 (Robot Operating System)  and  PlanSys2 , a dedicated planning framework for ROS 2. 9 6.1 PlanSys2 Architecture PlanSys2 manages the lifecycle of the PDDL plan. It consists of: Domain Expert:  Parses warehouse_drone_continuous.pddl. Problem Expert:  Holds the current state (Knowledge Base). Planner:  Wraps a solver (e.g., POPF) to generate the plan. Executor:  Dispatches actions to the drone. 6.2 The Runtime Workflow Trigger:  User sends command "Scan the red box." VLM Node: Captures Image. Runs Inference (PaliGemma + Depth Net). Computes 3D coordinates. Calls PlanSys2 service add_instance("red_box", "target"). Calls PlanSys2 service update_function("tx", "red_box", 12.5). Sets goal (scanned red_box). Planning:  PlanSys2 calls POPF. POPF returns: fly_to_target(drone, red_box) scan_target(drone, red_box) Action Execution:  The fly_to_target action is implemented as a C++ Behavior Tree node. 26 It queries the Problem Expert for the values of (tx red_box), (ty red_box), (tz red_box). It sends a trajectory command to the drone's flight controller (e.g., PX4) via the mavros/setpoint_position/local topic. It monitors the odometry topic. When distance < threshold, it returns SUCCESS. 6.3 Handling Failures If the perception is noisy and the drone flies to the wrong location, the scan_target action (which might utilize a secondary visual check) will fail. PlanSys2 handles this via  Replanning . The system updates the drone's position in the Knowledge Base and triggers a new planning cycle, potentially triggering a "search" behavior if defined in the domain. 7. Conclusion This report has defined a complete neuro-symbolic architecture for autonomous drone navigation in continuous warehouse environments. By rejecting the rigidity of graph-based waypoints in favor of  PDDL 2.1 Numeric Fluents , and by leveraging  Fine-Tuned Vision Language Models  as structured state estimators, we achieve a system capable of interpreting high-level instructions and executing precise 3D maneuvers. The key technical insights are: Dynamic Instantiation:  Treating physical locations not as static symbols but as dynamic numeric variables populated at runtime. Structured Fine-Tuning:  Constraining VLM output to JSON schemas to guarantee PDDL syntax validity. Hybrid Perception:  Combining semantic understanding (VLM) with geometric reconstruction (Depth Nets) to bridge the pixel-to-predicate gap. This approach transforms the drone from a passive follower of pre-defined paths into an active, reasoning agent capable of adapting to the dynamic reality of modern logistics.
